4. Wiggum Loop — Autonomous Optimization

Two Modes
Local Mode (wiggum.sh)
Developer's Machine
Uses Claude Code CLI (claude -p)
Each invocation = fresh agent, zero memory
Git branch is the only state

./wiggum.sh --instance oil-gas-leases
Cloud Mode (wiggum_cloud.py)
GitHub Actions / Any Python Runtime
Uses Claude API calls directly (no CLI)
Each API call = fresh instance, no context
Pushes improved code back to branch

uv run wiggum_cloud.py --cycles 10 
         ┌─────────────────────────────────────────────┐
         │           THE WIGGUM LOOP                   │
         │                                             │
         │  1. Read current process.py + eval results  │
         │  2. Call Claude: "fix the worst fields"     │
         │  3. Apply changes to process.py             │
         │  4. Run evaluate.py                         │
         │  5. Accuracy up? → git commit (KEEP)        │
         │     Same or down? → git reset (DISCARD)     │
         │  6. Spawn fresh agent → go to 1             │
         │                                             │
         │  Code accumulates intelligence.             │
         │  Agents are stateless.                      │
         └─────────────────────────────────────────────┘

7. Component Inventory

Component Where It Lives What It Does
React Frontend App Service static-build/ UI — upload PDFs, review extractions, manage schemas, configure DB. Auth0 SPA SDK handles login.
FastAPI Backend App Service app.py 15 API endpoints. JWT validation, org-scoped data, orchestrates Doc Intel + Claude.
Auth Module App Service auth/ Decoupled auth — JWT validation, RBAC, org-scoped path resolution. 4 files, ~250 lines.
Extraction Pipeline App Service process.py The file the Wiggum loop modifies. Pre-process → LLM extract → post-process.
SQL Generator App Service sql_gen.py JSON → SQL INSERT statements. Supports SQL Server, PostgreSQL, MySQL.
Doc Intel Wrapper App Service doc_intel.py Azure Document Intelligence SDK wrapper. Caches results per org.
Evaluation Harness App Service evaluate.py Fixed metric. Compares extraction output to ground truth. Field-level accuracy.
Wiggum Loop (local) Developer Machine wiggum.sh — spawns fresh Claude Code agents in a bash loop.
Wiggum Loop (cloud) GitHub Actions wiggum_cloud.py — pure Python, calls Claude API directly. No CLI needed.
Instance Manager Developer Machine instance.sh — create/switch/sync client branches for multi-tenant extraction.
CI/CD Deploy GitHub Actions deploy.yml — builds React, deploys to App Service on push to main.
Auth0 External SaaS Organizations, RBAC, login, JWT issuance. 4 roles: admin, dev, business, viewer.
Azure Doc Intel External SaaS OCR + layout analysis for PDFs. prebuilt-layout model.
Claude API External SaaS LLM for structuring raw extraction into schemas. Also used by Wiggum loop.

8. Do We Need More Infrastructure?

Current setup: everything runs on ONE Azure App Service.

The React build and FastAPI backend are deployed together. FastAPI serves the static files AND handles API requests. This is fine for the current scale. You do NOT need a separate Container App or Function App yet.

When to split:
TriggerAction
Frontend needs a CDN / faster global delivery Move React build to Azure Static Web Apps (free, CDN-backed). Point it at the App Service API.
Backend needs auto-scaling for heavy extraction load Move FastAPI to Azure Container App (scales to zero, auto-scales up).
Long-running extractions timeout on App Service Move extraction to Azure Container Instance or a background queue (Azure Service Bus).
File storage fills App Service disk Swap local filesystem for Azure Blob Storage (per-org containers).
Multiple teams working on frontend vs backend Split into two repos. Frontend deploys to Static Web Apps, backend stays on App Service.
Bottom line: You're correctly deployed today. The architecture is ready to split but doesn't need to yet. When you hit one of the triggers above, the split is a config change — no code rewrite.
